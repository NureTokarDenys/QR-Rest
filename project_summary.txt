PROJECT OVERVIEW
────────────────
QR-based restaurant management system. Guests scan a QR code at the table,
browse the menu, customise dishes, place orders and track order status in
real time. Staff (waiters, cooks, admins) manage orders via a separate staff
interface. Built as a SPA on a Node.js / Express backend with MongoDB.

USER ROLES & ACCESS
================================================================================

guest         — Anonymous user at a table (session via QR scan)
                Routes: /menu, /category/:id, /dish/:id, /cart, /confirm,
                        /order-status[/:orderId], /profile, /order-history,
                        /restaurant-reviews

waiter        — Table service staff
                Routes: /staff/map, /staff/table/:id, /staff/orders,
                        /staff/order/:id, /staff/settings

cook          — Kitchen staff
                Routes: /staff/cooking, /staff/orders, /staff/order/:id,
                        /staff/menu (+ category/new, category/:id, dish/new,
                        dish/:id, pdf), /staff/extras
                        (+ /:type/new, /:type/:id), /staff/settings

waiter_cook   — Combined role (union of waiter + cook routes,
                plus /staff/map and /staff/menu)

admin         — Full access to all staff routes plus:
                /staff/analytics (RequirePlan — premium only)
                /staff/restaurant-settings
                /staff/staff
                /staff/reviews (RequirePlan — premium only)

root_admin    — First admin created during onboarding. Treated as `admin`
                by ProtectedRoute (the route guard expands ['admin'] to
                ['admin','root_admin']). Cannot be deactivated or have
                role changed by other admins.

ProtectedRoute (with the convenience wrapper <Guard>) enforces role checks
and redirects unauthorised users to /forbidden?required=…&from=….
ClientOnlyRoute redirects authenticated staff users back to /staff/map so
client URLs never render staff inadvertently.

FREEMIUM MODEL
================================================================================

PLAN FIELD (backend)
  Restaurant.plan: 'free' | 'premium'  (default: 'free')
  Exposed in: public /restaurants listing, /qr/:shortCode response,
              /:restaurantId/info response

BACKEND GATING — requirePlan middleware
  File: QR Rest API/src/middleware/requirePlan.js
  Usage: router.post('/route', requirePlan('premium'), handler)
  Error response on mismatch:
    { error: { code: 'PLAN_REQUIRED', requiredPlan, currentPlan }, meta: {} }
  Applied to (routes/):
    All /admin/analytics routes                — premium only
    All /admin/reviews routes                  — premium only
    POST /admin/translations/auto-translate    — premium only
    All /admin/translate routes                — premium only
    POST /reviews/restaurant                   — premium only (guests)
    POST /reviews/dish                         — premium only (guests)
  LiqPay order payments (POST /:restId/payments/initiate) are gated by
  whether the restaurant has saved its own LiqPay keys (PAYMENT_NOT_CONFIGURED
  if not), independent of the requirePlan middleware.

BACKEND LIMITS — PLAN_LIMIT_REACHED error
  Code: 'PLAN_LIMIT_REACHED', fields: requiredPlan, limitType, limit
  Applied to:
    POST /admin/menu/categories       — free limit: 5 categories
    POST /admin/menu/items            — free limit: 50 items
    POST /admin/menu/items/:id/image  — free limit: 3 images per item
    POST /admin/staff                 — free limit: 3 staff accounts

FREE PLAN LIMITS SUMMARY
  Categories:      5
  Menu items:      50
  Images per dish: 3
  Staff accounts:  3   (admin counts toward the limit)
  Payment:         cash only (online via LiqPay needs setup + premium)
  Reviews:         disabled for guests and admin
  Analytics:       disabled
  PDF generator:   page accessible, modal-gated to premium
  Auto-translate:  disabled

PREMIUM PLAN
  All limits removed. All features unlocked.
  Single source of truth for pricing lives on the backend
  (routes/subscriptions.js: PREMIUM_PRICE = 800 UAH / month). UpgradeModal
  fetches the live price via GET /:restId/subscriptions/price.
  POST /:restId/subscriptions/initiate creates a LiqPay subscribe payment.

FRONTEND — PlanContext (src/context/PlanContext.jsx)
  Thin wrapper around StaffDataContext.restaurant — no separate fetch:
    const { restaurant } = useStaffData();
    plan = restaurant?.plan || 'free'; planLoading = restaurant === null;
  Exposed: { plan, isPremium, planLoading }
  This means plan switches via WS (RESTAURANT_UPDATED) propagate instantly.
  Hook: usePlanContext() — raw context access.

FRONTEND — AppContext plan tracking (client-side)
  AppContext.restaurantPlan — mirrors plan for guest/client flows
  Sources:
    • QR scan (initSession): reads restaurantPlan from /qr response
    • Restaurant info restore: reads plan from /:restId/info
    • selectRestaurant (picker): reads plan from restaurant listing
  Persisted to: localStorage key 'restaurantPlan'

FRONTEND — usePlan hook (src/hooks/usePlan.js)
  Universal hook used by both staff and client pages.

  Resolution order: PlanContext → AppContext.restaurantPlan → 'free'

FRONTEND — RequirePlan route guard (src/components/RequirePlan/index.jsx)
  Wraps a route element; renders null while plan is loading, redirects free
  users to /staff/map. Applied in App.jsx to /staff/analytics and
  /staff/reviews so the pages never even mount on the free plan.

FRONTEND — UpgradeModal component (src/components/UpgradeModal/)
  Props: open, onClose, reason (optional i18n key), ns (default 'components')
  Shows: premium badge, title, optional reason text, 7 feature bullets,

MULTI-LANGUAGE SYSTEM
================================================================================

FRONTEND LANGUAGE CODES
  'ua' → maps to ISO 'uk' when sent to backend (Accept-Language header)
  'en' → stays 'en'
  Backend also supports pl / de / fr / cs / sk (entries commented out in
  langs.js until matching locale JSONs are added).
  Helper: toApiLang('ua') → 'uk',  fromApiLang('uk') → 'ua'
  SUPPORTED_LANGS[0] is the SOURCE language (Ukrainian = 'ua')

I18N SETUP (src/i18n/index.js)
  i18next with react-i18next. Language stored in localStorage 'lang'.
  Namespaces loaded per page (e.g. 'menu', 'staffSettings', 'cooking',
  'pdfGenerator', 'components', …).
  Restaurant default language read from localStorage 'restaurantDefaultLang'.

KEY HELPERS (src/i18n/langs.js)
  fieldFor(base, lang)
    Returns the correct field key for a given language.
    fieldFor('name', 'ua') → 'name'       (source lang = no suffix)
    fieldFor('name', 'en') → 'name_en'
    Used everywhere multilingual form fields need to map to an object key.

  emptyI18n(...bases)
    Creates a blank multilingual object for form initialization.
    emptyI18n('name', 'description') →
      { name: '', name_en: '', description: '', description_en: '' }

  SUPPORTED_LANGS — array of { code, label, flag, apiCode } objects
  SOURCE_LANG — first entry, currently 'ua'

DATABASE TRANSLATION STRUCTURE
  Two storage patterns exist across the data model:

  Pattern A — flat fields (read directly by the frontend):
    { name: "Борщ з пампушками",   name_en: "Borsch with buns",
      description: "Класичний...", description_en: "Classic..." }

  Pattern B — translations object (driven by setTranslationEntry):
    { translations: {
        uk: { name: { value: "...", isManual: true } },
        en: { name: { value: "...", isManual: false } }
      }
    }
  Backend uses 'uk' as the ISO code for Ukrainian in the translations object.
  Auto-translation is scheduled by translationService.scheduleAutoTranslate
  on every admin mutation (premium plan only) and writes Pattern B.

  IMPORTANT — which entities store which pattern (there is NO automatic
  Pattern B → Pattern A sync; do not assume flat name_en always exists):
    • Top-level docs — MenuItem, Category, Ingredient, AddOn, ComponentGroup,
      ComponentOption — store ONLY `name` + `translations` (Pattern B). They do
      NOT have a flat top-level `name_en`.
    • MenuItem EMBEDDED sub-schemas (ingredients[], addons[],
      componentGroups[], options[]) DO carry flat `name` + `name_en`.
    • OrderItem snapshots freeze ONLY the source-language `name`
      (menuItemName, excludedIngredients[].name, addons[].name,
      componentGroupChoices[].groupName/optionName) — no `_en`.

  Because of the above, endpoints that return order / menu / extras data must
  EXPLICITLY surface `name_en`. Two backend helpers do this so the frontend
  always receives flat name + name_en:
    • i18nOrderEnricher.enrichOrderItems(items) — joins MenuItem (by id) and
      rewrites each item to { menuItemId:{_id,name,name_en}, excludedIngredients
      [{_id,name,name_en}], addons[...name_en], componentGroupChoices
      [{ groupId:{_id,name,name_en}, optionId:{_id,name,name_en} }] }.
      Used by GET /kitchen/orders, GET /orders/:id, create-order and
      add-items responses.
    • i18nFlatten.flattenTranslations(doc, fields) — copies
      translations.<lang>.<field>.value into flat <field>_<lang> keys.
      Used by GET /admin/menu/categories, /items, /items/:id and the
      /admin/menu/extras summary.
  /admin/tables already inlined this join for the table-map dish list.

RENDERING MULTILINGUAL TEXT IN THE UI
  Preferred: the useLocalField() hook (src/i18n/useLang.js) — returns a
  localiser bound to the active i18next language:
    const local = useLocalField();
    local(dish, 'name')        // dish.name (ua) | dish.name_en (en) | source
  It re-renders reactively on language change (built on useTranslation), so NO
  data refetch is needed — the backend already ships both name and name_en in a
  single response. Do NOT re-fetch order/menu data just because the language
  changed.
  Lower-level alternative when you only need the key:
    const key  = fieldFor('name', i18n.language);   // 'name' | 'name_en'
  All staff order/menu pages — Cooking, WaiterOrders, OrderDetail, TableDetail,
  MenuManagement, ExtrasManagement, Analytics (+ KanbanCard, TableDishList,
  MenuDishRow, TopDishRow, TopCategoryItem) — now use useLocalField().
  Serving-group names are intentionally NOT localised (shown verbatim).

ACCEPT-LANGUAGE HEADER
  Axios request interceptor (api/client.js) auto-sets Accept-Language using
  toApiLang(lang) so the backend returns translated content for endpoints
  that support per-request localisation.

WEBSOCKET SYSTEM
================================================================================

CONNECTION LIFECYCLE
  ONE shared WebSocket lives in AppContext (no per-page sockets). The
  useWebSocket hook is now a thin wrapper that registers / unregisters a
  listener on the global connection.

  URL priority: VITE_WS_URL → swap http→ws from VITE_API_URL → same-origin
                /ws.  Auth via querystring: ?token=<jwt>&session_token=<…>

  Reconnection: max 5 retries, 3-second delay,
                status: idle → connecting → connected → reconnecting → failed
  Keepalive:    PING every 25 s → PONG response
  Replay:       On reconnect sends REPLAY_REQUEST with last_event_id.
                Server keeps a 10-minute per-room event queue (wsService.js).

SUBSCRIPTION ROOMS
  session:<sessionToken>       — guest client (subscribed on QR scan)
  table:<tableId>              — table-level events (table card lookups)
  restaurant:<restaurantId>    — staff users get all restaurant events
                                 here (joined automatically on auth)
  kitchen:<restaurantId>       — cook / kitchen events (ORDER_NEW,
                                 ORDER_UPDATED, ORDER_COMPLETED,
                                 DISH_STATUS_UPDATED, GROUP_STATUS_UPDATED…)
  waiter:<restaurantId>        — waiter-specific events (ORDER_UPDATED,
                                 ORDER_COMPLETED, WAITER_CALL, WAITER_CALL_CASH,
                                 WAITER_CALL_RESOLVED…)

SERVER → CLIENT EVENTS
  ORDER_NEW, ORDER_UPDATED, ORDER_VOID, ORDER_CANCELLED, ORDER_COMPLETED,
  ORDER_ITEMS_ADDED, DISH_STATUS_UPDATED, GROUP_STATUS_UPDATED,
  TABLE_CREATED, TABLE_UPDATED, TABLE_DELETED, TABLES_REORDERED,
  TABLE_STATUS_UPDATED,
  MENU_UPDATED, EXTRAS_UPDATED, STAFF_CREATED, STAFF_UPDATED,
  RESTAURANT_UPDATED,
  WAITER_CALL, WAITER_CALL_CASH, WAITER_CALL_RESOLVED,
  PAYMENT_COMPLETED,
  NOTIFICATION_NEW           (guest order notifications)
  PONG                       (keepalive latency probe)

NOTIFICATION_NEW PAYLOAD
  { type, title_uk, title_en, body_uk, body_en, data: {} }
  NotificationType: dish_status_updated | order_cancelled | items_added |
                    payment_completed_cash | payment_completed_epay

CLIENT-SIDE WS LISTENERS
  AppContext: addWsListener(fn) / removeWsListener(fn) / wsSubscribe(room)
  Pages register custom handlers for specific events; the AppContext routes
  every inbound message to every registered listener.

NOTABLE IMPLEMENTATION NOTES
================================================================================

PDF GENERATOR (/staff/menu/pdf)
  • Settings rail (template presets, page format, orientation, margins,
    language, font family / size, colours, content toggles, image options,
    drag-orderable categories with per-dish checkboxes).
  • Off-screen measurement layer renders every flat item at the real content
    width; pagination is driven by offsetHeight + ResizeObserver, with an
    integrated orphan-header fix.
  • Bilingual layouts: stacked (UA then EN) or two columns.
  • Download: jsPDF + html2canvas — captures every page article at 2× scale
    and saves directly as `menu.pdf`.  No browser print dialog.
  • Free plan: UpgradeModal opens immediately on page load.
  • Server pagination cap is 9999 for the admin /menu/items list so the
    full dish set is fetched in one request.

ONE ACTIVE ORDER PER ACCOUNT
  POST /:restId/orders checks for any non-terminal order belonging to the
  same account across ALL restaurants — not just the current table/restaurant.
  Authenticated users are keyed by userId; guests are keyed by sessionToken.
  Terminal statuses (excluded from the check): cancelled, completed_cash,
  completed_epay.
  Error code: ACTIVE_ORDER_EXISTS (HTTP 409).
  Response includes activeOrderId and activeOrderRestaurantId so the client
  can redirect to the existing order.
  ConfirmOrder/index.jsx catches both ACTIVE_ORDER_EXISTS and the legacy
  TABLE_ORDER_LOCKED code and shows a lockedBanner with a "View order" link.

ORDER RESTORATION
  3-strategy fallback on login (cache → known order id → user orders scan).
  The active order's restaurantPlan is preserved so the review UI gates
  correctly even after a refresh.

LIQPAY INTEGRATION
  Per-restaurant LiqPay keys are stored encrypted (encryptionService) and
  read by /payments/initiate. The platform-level LiqPay service is used for
  subscription checkout (premium upgrade).

GOOGLE OAUTH
  Login: /api/auth/google → /auth/callback (handled by OAuthCallback page)
  Link existing account: initiateGoogleLink stores a short-lived
                          google_link_nonce cookie before redirecting.
  Unlink: only allowed when the account also has a password.

ROUTING REDIRECTS
  RootRedirect chooses /menu vs /restaurants based on sessionToken /
  restaurantId presence; waits for AppContext order-restoration to finish.

IMPORTANT PATTERNS / CONVENTIONS
================================================================================

• Module CSS class names are camelCase: styles.myClassName
• i18n keys are snake_case: t('my_translation_key')
• All restaurant-scoped API calls use restaurantId from AppContext /
  getStoredRestaurantId().
• Money values are floats (UAH).
• publicId is the external 8-char restaurant identifier; MongoDB _id is
  internal only.
• Serving groups are always present (minimum 1 per order — 'main').
• "Snapshot" fields (menuItemName, unitPrice, ingredient / addon /
  componentGroup names) are frozen on the OrderItem at order-creation and
  never mutate even if the menu is updated later. NOTE: snapshots freeze only
  the SOURCE-language name — there is no name_en on the OrderItem document.
• name_en for order items is NOT stored on the snapshot; it is injected at
  response time by i18nOrderEnricher (live MenuItem join). So endpoints that
  return order items must run them through enrichOrderItems — otherwise the
  frontend only sees the Ukrainian name and language switching silently fails.
• usePlan() is the universal hook for plan-gated logic (staff + client).
• UpgradeModal is the universal component for all premium upsell flows.
• RequirePlan is preferred over per-page modals when an entire page should
  not be reachable on the free plan.
• `root_admin` is treated as `admin` by ProtectedRoute (auto-included when
  baseRoles contains 'admin').
• All staff features rely on the single global WebSocket; do not open new
  sockets — use useWebSocket({ onMessage, enabled }) instead.
